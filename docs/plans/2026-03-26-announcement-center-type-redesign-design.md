# 公告中心短期/长期公告重构设计

日期：2026-03-26

## 背景

当前公告中心后台已经具备公告创建、编辑、发布、下线、删除和 App 拉取能力，但编辑页直接暴露了大量内部字段：

- 后台有英文选项，非技术使用者难以理解。
- “今日不提醒”“强提醒”“开始时间/结束时间”等能力没有形成清晰的业务语义。
- 用户需要明确区分“短期公告”和“长期公告”，并让 App 的关闭行为随公告类型变化。

## 目标

本次重构目标如下：

1. 在数据层显式支持公告类型，而不是只靠布尔字段和时间字段推断。
2. 后台编辑页改为中文业务表单，让每个选项都能直接看懂。
3. 支持短期公告与长期公告的差异化规则。
4. 保持与现有 App 接口兼容，同时补充更明确的类型信息。
5. 不重做当前公告弹窗视觉样式，只调整行为和按钮文案。

## 方案选择

采用 B 方案：新增显式字段 `announcement_type`。

原因：

- 类型语义落库后，后台、接口、App 都能围绕同一业务概念实现。
- 后续如果要扩展“活动公告”“系统通知”等分类，会比布尔字段组合更稳定。
- 能避免后台表单把已有底层字段硬塞给用户。

## 数据模型

在 `announcements` 表新增字段：

- `announcement_type VARCHAR(16) NOT NULL DEFAULT 'long'`

合法值：

- `short`：短期公告
- `long`：长期公告

规则：

- 短期公告必须有 `end_at`，通常也建议有 `start_at`。
- 长期公告允许 `end_at` 为空。
- 发布时，仍保持“同一时间只有一条 published 公告”的现有策略。

## 后台表单设计

编辑页调整为面向业务的 4 个区域：

1. 基础内容
- 公告标题
- 简介
- 正文
- 封面图

2. 展示方式
- 公告类型：短期公告 / 长期公告
- 公告等级：普通 / 成功 / 提醒 / 重要
- 弹窗方式：普通弹窗 / 强提醒弹窗
- 适用版本：正式版 / 开发版 / 全部
- App 版本范围

3. 发布时间
- 保存方式：存为草稿 / 立即发布 / 下线
- 开始时间
- 结束时间
- 根据类型显示对应说明

4. 跳转按钮
- 主按钮文案与链接
- 次按钮文案与链接

设计原则：

- 页面全部中文化。
- 把“内部值”隐藏在后端映射里，不让用户直接看到。
- 把会联动的选项写清楚说明，不要求用户理解底层实现。

## 业务行为

### 短期公告

- 后台要求必须填写结束时间。
- App 不显示“今日不提醒”。
- App 关闭按钮文案改为“`不再提示`”，调用现有永久确认逻辑。
- 若配置了跳转按钮，则仍显示跳转操作。

### 长期公告

- 可不填结束时间。
- App 允许显示“今日不提醒”。
- 主关闭按钮文案保留为“`知道了`”或“`我知道了`”。

## 接口兼容策略

现有接口 `/api/app/announcements/active` 保持原字段不变，并新增：

- `announcement_type`

兼容原则：

- 老字段 `allow_snooze_today` 仍返回，值由公告类型和配置共同决定。
- `force_popup` 继续保留。
- 新 App 使用 `announcement_type` 做更明确的按钮文案判断。

## App 侧改动

App 不改整体弹窗视觉，只调整交互语义：

- 数据模型增加 `announcementType`
- 短期公告：
  - 不显示“今日不提醒”
  - 关闭确认按钮显示“`不再提示`”
- 长期公告：
  - 允许显示“今日不提醒”
  - 关闭确认按钮显示“`知道了`”或“`我知道了`”

## 风险与兼容

主要风险：

- 现有数据库没有 `announcement_type` 字段，需要做启动时增量迁移。
- 已有数据没有类型，需要设置合理默认值。

处理方式：

- `ensureSchema()` 中补充 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 风格兼容逻辑。
- 历史数据默认视为 `long`，避免影响已上线公告。

## 验证范围

需要验证：

1. 新建短期公告可保存、可发布、必填结束时间生效。
2. 新建长期公告可保存、可发布、可不填结束时间。
3. 编辑历史公告不会报错。
4. 列表页能看出公告类型。
5. App 编译通过，且新字段解析正常。
