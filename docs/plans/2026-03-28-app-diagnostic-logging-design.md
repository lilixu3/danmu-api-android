# App 诊断日志与控制台聚合设计

日期：2026-03-28

## 背景

当前项目已经明确禁用了工作目录中的文件日志，核心运行日志统一通过 `/api/logs` 拉取并展示。这样做的优点是避免普通模式、Root 模式和不同工作目录下出现多份日志文件，但也带来了一个明显问题：

- App 自己的启动链路日志没有进入控制台。
- 当核心与 App 包装层不兼容、导致核心尚未成功启动时，`/api/logs` 为空。
- 这类故障在 `Application`、`BootReceiver`、`NodeService`、无障碍保活和 Root 启动阶段最常见，当前只能依赖 Logcat，不利于用户反馈问题。

## 目标

本次改造目标如下：

1. 为 App 进程新增独立的诊断日志能力，不再依赖核心 `/api/logs` 承载 App 启动故障。
2. 保持工作目录下日志文件不会无限增长，也不把 App 日志重新写回工作目录。
3. 继续保留现有 `/api/logs` 作为核心日志来源，不恢复 `danmuapi.log` 文件模式。
4. 控制台页面仍然只有一个入口，但可以同时查看核心日志、App 诊断日志和 Root 启动引导日志。
5. 清理日志时同时清掉内存日志与受控落盘日志，避免用户误以为已清空但磁盘仍残留旧记录。

## 方案选择

采用 C 方案：核心日志与 App 诊断日志分仓存储，在 UI 层统一聚合展示。

原因：

- 不破坏当前“核心日志只走 `/api/logs`”的既有设计。
- App 日志可在核心未启动成功时仍然留存，适合排查启动失败。
- 文件滚动和存储位置可完全由 App 控制，不受工作目录切换影响。

不采用的方案：

- 直接恢复工作目录文件日志：会把旧的日志膨胀问题、目录切换问题和普通/Root 多份日志问题重新带回来。
- 只把 `Logcat` 改成内存 `addLog()`：进程崩溃或跨进程启动故障时仍然容易丢失。

## 存储设计

### App 诊断日志

- 存储目录：`context.noBackupFilesDir/app-logs/`
- 主文件：`app.log`
- 轮转文件：`app.log.1`、`app.log.2`
- 单文件上限：`512KB`
- 总上限：约 `1.5MB`

设计原则：

- 不放在用户可切换的工作目录。
- 不参与系统自动备份，避免无意义地把诊断日志同步到云端。
- 小容量滚动即可满足反馈问题，不追求长期留存。

### Root 启动引导日志

- 存储目录：同 `app-logs/`
- 主文件：`root-bootstrap.log`
- 轮转文件：`root-bootstrap.log.1`
- 单文件上限：`128KB`

用途：

- 只接 Root 启动脚本阶段的 stdout/stderr。
- 用于弥补 Root 模式启动失败但 `/api/logs` 尚未可用时的排障盲区。

## 数据模型

现有 `LogEntry` 仅包含时间、级别和消息，无法区分来源。改造后扩展如下：

- `source`：日志来源，至少包含 `core`、`app`、`root_bootstrap`
- `tag`：日志标签，如 `NodeService`、`BootReceiver`

兼容原则：

- 现有核心 `/api/logs` 解析出的日志默认标记为 `core`
- 现有 `runtimeRepo.addLog()` 生成的内存事件日志默认标记为 `app`
- 控制台默认按时间合并，不要求用户理解底层来源

## App 日志写入方式

新增一个应用级单例日志器，提供以下能力：

1. 同时写入 Android `Logcat`
2. 将关键日志按结构化行写入 `app.log`
3. 根据文件大小自动轮转
4. 提供读取最近日志与清空日志的方法
5. 支持将 `Throwable` 压缩为单条可读消息并附加堆栈摘要

日志器不替代所有普通业务提示，只接管诊断价值高的启动链路日志。

## 接入范围

首批接入以下位置：

- `DanmuApiApplication`
- `RuntimeWarmupCoordinator`
- `BootReceiver`
- `MyPackageReplacedReceiver`
- `KeepAliveAccessibilityService`
- `NodeService`
- `RootRuntimeController`

原则：

- 保留原有 `Logcat` 输出能力。
- 同步写入 App 诊断日志。
- 对真正影响排障的失败、超时、恢复、降级、启动状态变化重点埋点。

## 控制台聚合策略

控制台的最终数据由三部分组成：

1. 核心 `/api/logs`
2. App 诊断日志文件
3. 现有 `_eventLogs` 内存事件日志

聚合规则：

- 按时间戳排序
- 统一套用现有日志条数上限
- 支持按来源筛选
- 复制日志时保留来源、标签和级别

工具页日志预览仍沿用现有数据源，不新增第二套预览逻辑。

## 清理策略

“清除日志”操作应同时执行：

1. 清空核心 `/api/logs`
2. 清空 `_eventLogs`
3. 删除 `app.log*`
4. 删除 `root-bootstrap.log*`

这样用户可以得到真正一致的“已清空”结果。

## 风险与兼容

主要风险：

- 如果接入方式过重，可能把大量普通调试日志写入文件，造成日志噪声。
- 如果直接全局劫持 `android.util.Log`，实现复杂且难控。

处理方式：

- 首批只接入关键启动链路，不做全局劫持。
- 新日志器以手动调用为主，优先替换最关键的 `Log.w` / `Log.e` / `Log.i`。
- 文件容量严格限制并滚动，避免无限增长。

## 验证范围

需要验证：

1. 普通模式启动成功时，控制台能同时看到核心日志和 App 启动日志。
2. 核心启动失败时，控制台仍能看到 App 侧诊断日志。
3. Root 模式启动超时或失败时，能看到 `root-bootstrap` 相关日志。
4. 清除日志后，控制台列表清空，磁盘中的诊断日志文件也被清掉。
5. 日志文件在反复写入后仍保持轮转，不会无限增长。
